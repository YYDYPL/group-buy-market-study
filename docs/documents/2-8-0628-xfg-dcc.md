# 2-8-0628-xfg-dcc 学习文档

## 一、分支目标

当前分支在上一阶段“人群标签节点过滤”的基础上，新增了 DCC 动态配置中心能力，用于在应用运行期间动态调整业务开关。

本分支的核心目标是：通过 HTTP 接口发布配置变更，借助 Redis Topic 广播消息，再由应用内的监听器刷新被 `@DCCValue` 标记的字段，使业务逻辑可以即时读取最新配置。

当前接入的业务场景是首页拼团试算的降级和切量：

1. `downgradeSwitch`：拼团活动降级开关，开启后直接拦截试算。
2. `cutRange`：按用户 ID 哈希做切量控制。

### 2. 新增 DCC 注解

新增文件：

- `grouph-buy-market-study-types/src/main/java/com/hjs/study/types/annotation/DCCValue.java`

注解定义：

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface DCCValue {
    String value() default "";
}
```

它用于标记需要动态配置的字段，例如：

```java
@DCCValue("downgradeSwitch:0")
private String downgradeSwitch;
```

配置格式是：

```text
配置名:默认值
```

## 三、DCC 发布接口

新增文件：

- `grouph-buy-market-study-trigger/src/main/java/com/hjs/study/trigger/http/DCCController.java`

`DCCController` 实现 `IDCCService`，提供 HTTP 接口：

```java
@RequestMapping(value = "update_config", method = RequestMethod.GET)
public Response<Boolean> updateConfig(String key, String value)
```

完整访问路径：

```text
/api/v1/gbm/dcc/update_config?key=downgradeSwitch&value=1
```

接口执行逻辑：

```java
dccTopic.publish(key + "," + value);
```

也就是说，Controller 不直接修改业务 Bean，而是把配置变更发布到 Redis Topic。

成功时返回：

```java
ResponseCode.SUCCESS
```

异常时返回：

```java
ResponseCode.UN_ERROR
```

## 四、DCCValueBeanFactory 实现

新增文件：

- `grouph-buy-market-study-app/src/main/java/com/hjs/study/config/DCCValueBeanFactory.java`

这是本分支最核心的实现类。它同时承担两类职责：

1. 启动时扫描 `@DCCValue` 字段，初始化配置值。
2. 运行时监听 Redis Topic，刷新 Redis 和 Bean 字段。

### 1. 配置 key 规范

统一配置前缀：

```java
private static final String BASE_CONFIG_PATH = "group_buy_market_dcc_";
```

例如：

```text
downgradeSwitch -> group_buy_market_dcc_downgradeSwitch
cutRange        -> group_buy_market_dcc_cutRange
```

### 2. 启动时扫描注解字段

`DCCValueBeanFactory` 实现了 `BeanPostProcessor`：

```java
public class DCCValueBeanFactory implements BeanPostProcessor
```

在 `postProcessAfterInitialization` 中，它会遍历 Bean 字段：

```java
if (!field.isAnnotationPresent(DCCValue.class)) {
    continue;
}
```

然后解析注解值：

```java
String[] splits = value.split(":");
String key = BASE_CONFIG_PATH.concat(splits[0]);
String defaultValue = splits.length == 2 ? splits[1] : null;
```

初始化逻辑：

1. 如果 Redis 中不存在该配置 key，则写入默认值。
2. 如果 Redis 中已经存在该配置 key，则读取 Redis 中的最新值。
3. 通过反射把值设置到当前 Bean 字段。
4. 把配置 key 和 Bean 对象放入 `dccObjGroup`，方便后续监听器定位要更新的对象。

### 3. 处理 AOP 代理对象

为了兼容 Spring AOP 代理，代码没有直接使用 `bean.getClass()` 完成全部逻辑，而是判断代理对象：

```java
if (AopUtils.isAopProxy(bean)) {
    targetBeanClass = AopUtils.getTargetClass(bean);
    targetBeanObject = AopProxyUtils.getSingletonTarget(bean);
}
```

这样可以拿到真实目标类和目标对象，避免因为代理类结构变化导致读取不到自定义注解。

### 4. Redis Topic 监听

`DCCValueBeanFactory` 注册了一个名为 `dccTopic` 的 Bean：

```java
@Bean("dccTopic")
public RTopic dccRedisTopicListener(RedissonClient redissonClient)
```

Topic 名称：

```text
group_buy_market_dcc
```

监听到消息后，按逗号拆分：

```java
String[] split = s.split(Constants.SPLIT);
String attribute = split[0];
String value = split[1];
```

然后：

1. 拼接 Redis key：`group_buy_market_dcc_` + `attribute`。
2. 如果 Redis key 不存在，则忽略。
3. 如果 Redis key 存在，则更新 Redis 值。
4. 从 `dccObjGroup` 找到对应 Bean。
5. 通过反射更新 Bean 字段。

这样就完成了从外部请求到内存字段的动态刷新。

## 五、DCC 配置承载类

新增文件：

- `grouph-buy-market-study-infrastructure/src/main/java/com/hjs/study/infrastructure/dcc/DCCService.java`

该类定义了当前业务使用的两个动态配置项：

```java
@DCCValue("downgradeSwitch:0")
private String downgradeSwitch;

@DCCValue("cutRange:100")
private String cutRange;
```

### 1. 降级开关

```java
public boolean isDowngradeSwitch() {
    return "1".equals(downgradeSwitch);
}
```

含义：

- `0`：关闭降级。
- `1`：开启降级。

### 2. 切量范围

```java
public boolean isCutRange(String userId)
```

实现逻辑：

1. 计算 `userId.hashCode()` 的绝对值。
2. 取最后两位：`hashCode % 100`。
3. 判断最后两位是否小于等于 `cutRange`。

如果 `cutRange = 100`，基本表示全量放开。

如果 `cutRange = 10`，则大约只放开 10% 左右的用户。

### 3. SwitchNode 增加降级和切量判断

修改文件：

- `grouph-buy-market-study-domain/src/main/java/com/hjs/study/domain/activity/service/trial/node/SwitchNode.java`

`SwitchNode` 是试算责任链中的开关节点。本分支把 DCC 控制接入这里。

执行顺序：

```java
if (repository.downgradeSwitch()) {
    throw new AppException(ResponseCode.E0003.getCode(), ResponseCode.E0003.getInfo());
}

if (!repository.cutRange(userId)) {
    throw new AppException(ResponseCode.E0004.getCode(), ResponseCode.E0004.getInfo());
}

return router(requestParameter, dynamicContext);
```

也就是说：

1. 如果开启降级，直接拦截试算。
2. 如果用户不在切量范围内，也直接拦截试算。
3. 只有通过开关判断后，才继续进入 `MarketNode`。

## 十、完整调用链路

### 1. 配置更新链路

```text
DCCController.updateConfig(key, value)
  -> dccTopic.publish("key,value")
  -> Redis Topic group_buy_market_dcc
  -> DCCValueBeanFactory 监听消息
  -> 更新 Redis Bucket group_buy_market_dcc_key
  -> 通过反射更新 @DCCValue 字段
```

### 2. 业务读取链路

```text
SwitchNode
  -> repository.downgradeSwitch()
  -> ActivityRepository
  -> DCCService.isDowngradeSwitch()
  -> downgradeSwitch 字段最新值
```

切量链路类似：

```text
SwitchNode
  -> repository.cutRange(userId)
  -> DCCService.isCutRange(userId)
  -> cutRange 字段最新值
```

## 十一、当前实现边界

1. DCC 消息格式依赖 `key,value`，当前没有对消息格式做完整校验。
2. 监听器只在 Redis key 已存在时才更新；如果没有被 `@DCCValue` 初始化过，发布消息会被忽略。
3. `dccObjGroup` 以配置 key 映射 Bean，如果多个 Bean 使用同一个配置 key，当前实现会被后注册的 Bean 覆盖。
4. `DCCController.updateConfig` 返回的 `Response<Boolean>` 当前没有填充 `data` 字段。
5. 切量算法基于 `userId.hashCode() % 100`，适合简单灰度，不适合严格流量分桶统计。

## 十二、代码走读顺序建议

建议按以下顺序学习：

1. `IDCCService` 和 `Response`：理解 DCC 对外接口。
2. `DCCController`：看配置如何发布到 Redis Topic。
3. `DCCValue`：理解动态配置字段的标记方式。
4. `DCCValueBeanFactory`：重点阅读启动扫描、Redis 初始化、Topic 监听、反射更新字段。
5. `DCCService`：看当前业务配置项和判断方法。
6. `IActivityRepository` 与 `ActivityRepository`：看领域层如何通过仓储读取 DCC。
7. `SwitchNode`：看降级和切量如何拦截试算链路。
8. `DCCControllerTest`：看配置更新和业务试算联动验证。

## 十三、设计要点总结

1. `@DCCValue` 将业务配置字段和默认值声明在使用方附近，降低配置分散度。
2. `DCCValueBeanFactory` 使用 BeanPostProcessor 扫描字段，启动时完成默认值初始化。
3. Redis Bucket 保存配置值，Redis Topic 负责广播配置变更。
4. 运行时通过反射更新 Bean 字段，业务读取字段时即可拿到最新值。
5. `SwitchNode` 是接入动态配置的合适位置，因为它本来就是试算责任链中的开关节点。
6. DCC 能力先服务于降级和切量，后续可以继续扩展到活动开关、算法参数和灰度策略。
